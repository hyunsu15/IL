## 품질테스트

## 신뢰성 및 독립성

// 격리성은 검토 필요
- 격리성: 테스트가 실제 DB나 외부 API에 의존하지 않고 Mocking되어 있는가?
- 순서 독립성: 테스트 A가 실행된 후 B가 실행되어도 결과가 변함없는가? (공유 변수 사용 금지)

- 데이터 초기화: 테스트 실행 전후로 데이터(인메모리 DB 등)가 깨끗하게 정리되는가?

## 비즈니스 정합성
- 경계값 분석 (Boundary Value): $if$문의 조건식(>, >=, ==)에서 경계값(예: 0, 최대치, null)을 테스트하는가?
- Happy Path 외 검증: 성공 케이스뿐만 아니라 **예외 상황(Negative Case)**에 대한 테스트가 포함되었는가?
- 의미 없는 Assert 지양: 단순히 assertNotNull(object)만 하고 끝내지 않고, 객체 내부의 핵심 필드값이 예상과 일치하는지 검증하는가?

## 유지보수성
- 구현 세부사항 의존도: 내부 메서드 호출 횟수(verify)보다 **최종 상태(State)**를 검증하는가?

- 가독성 (Display Name): 테스트 이름만 보고도 "어떤 상황에서 어떤 결과가 나와야 하는지" 비즈니스 언어로 이해되는가?

- 테스트 코드 중복: 반복되는 셋업 로직은 @BeforeEach나 별도의 팩토리 메서드로 분리했는가?

## AI 특화 체크항목
- Logic Hallucination: AI가 존재하지 않는 필드나 메서드를 사용하여 컴파일 에러를 방지했는가?

- Tautology (동의어 반복): A를 넣으면 A가 나온다 식의 로직 없는 테스트인가? (예: set(10) 호출 후 바로 get() == 10 확인)

- Mocking 적절성: 복잡한 의존성을 가진 Bean들이 적절히 @MockitoBean 처리되어 테스트 속도가 느려지지 않는가?

- 테스트코드는 컴파일되야 한다.
